# ML-система предиктивной диагностики СВЧ-модулей

![Python](https://img.shields.io/badge/Python-3.12-blue)
![scikit-learn](https://img.shields.io/badge/scikit--learn-ML-orange)
![Streamlit](https://img.shields.io/badge/Streamlit-App-red)
![Status](https://img.shields.io/badge/status-MVP-green)

End-to-end ML-система для диагностики технического состояния СВЧ-модулей по температурным, вибрационным и электрическим временным рядам.

Проект моделирует производственный сценарий: СВЧ-модули проходят испытания на стендах, система анализирует сигналы датчиков, определяет техническое состояние оборудования, оценивает риск и формирует рекомендации для оператора и инженера-техника.

## Кратко

Проект объединяет два уровня:

1. ML-уровень: временные ряды → окна сигналов → признаки → модели → метрики → интерпретация.
2. Продуктовый уровень: состояние модуля → риск → предупреждение → план проверки.

| Параметр | Значение |
|---|---:|
| Тип задачи | Многоклассовая классификация |
| Количество классов | 7 |
| Количество экспериментов | 560 |
| Частота дискретизации | 100 Гц |
| Размер окна | 5 секунд |
| Количество окон | 12 880 |
| Количество признаков | 100+ |
| Лучшая модель | Extra Trees |
| Accuracy | ≈ 0.87 |
| F1-macro | ≈ 0.87 |

## Задача

На испытательном участке СВЧ-модули могут переходить в неисправные состояния постепенно. Перегрев, ухудшение охлаждения, повышенная вибрация и механический износ проявляются в динамике сигналов, а не в одной точке измерения.

Цель проекта — построить ML-систему, которая определяет состояние модуля по окну временного ряда и переводит результат модели в понятное производственное действие.

## Данные

В проекте используется синтетический датасет, построенный по инженерной логике работы СВЧ-модулей. Такой подход выбран из-за ограниченной доступности реальных промышленных данных по отказам радиоэлектронного оборудования.

Симуляция учитывает:

1. индивидуальные особенности модулей;
2. тепловое смещение;
3. эффективность охлаждения;
4. фоновую вибрацию;
5. шум и дрейф датчиков;
6. сдвиг резонансных частот;
7. нагрузку, ток и напряжение;
8. скорость вентилятора;
9. слабые неисправности;
10. переходные состояния;
11. пересечение классов.

## Feature Engineering

Модель обучается не на отдельных точках сигнала, а на признаках, рассчитанных по окнам временного ряда.

| Группа признаков | Примеры |
|---|---|
| Температурные | среднее, максимум, минимум, диапазон, тренд |
| Тепловая динамика | разность кристалл-корпус, доля времени выше порога |
| Вибрационные | RMS, максимум, пик-фактор, тренд, zero-crossing rate |
| Частотные | доминирующая частота, спектральный центр, энергия по диапазонам |
| Режимные | нагрузка, ток, напряжение, скорость вентилятора |
| Интегральные индексы | thermal stress index, vibration stress index, combined stress index |

## Модели

В проекте сравниваются несколько моделей: от baseline до ансамблевых алгоритмов.

| Модель | Роль |
|---|---|
| Logistic Regression | baseline |
| Random Forest | нелинейная ансамблевая модель |
| Extra Trees | лучшая итоговая модель |
| HistGradientBoosting | альтернативная модель бустинга |

Лучшая модель выбирается по F1-macro, так как задача многоклассовая и важно учитывать качество по всем состояниям.

## Результаты

| Метрика | Значение |
|---|---:|
| Accuracy | ≈ 0.87 |
| Precision macro | ≈ 0.87 |
| Recall macro | ≈ 0.87 |
| F1-macro | ≈ 0.87 |

Модель не показывает искусственно идеальные метрики. В данных есть шум, слабые неисправности и пересекающиеся классы, поэтому задача остаётся реалистичной.

Наиболее сложные пары классов:

1. перегрев и ухудшение охлаждения;
2. повышенная вибрация и износ механического узла;
3. норма и ранняя стадия ухудшения охлаждения.

## Запуск

Создание окружения и установка зависимостей:

~~~powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
~~~

Запуск полного ML-pipeline:

~~~powershell
python run_pipeline.py
~~~

Запуск Streamlit-приложения:

~~~powershell
python -m streamlit run .\app\main.py
~~~

После запуска приложение доступно по адресу:

~~~text
http://localhost:8501
~~~

## Ограничения

1. Данные являются синтетическими.
2. Система требует проверки на реальных промышленных данных.
3. Текущая версия решает задачу классификации известных состояний.
4. Для неизвестных дефектов нужен отдельный модуль anomaly detection.
5. Ролевая модель в интерфейсе является демонстрационной.
